# 记忆系统架构审计与重构建议

> 面向：交给 CC 施工。
> 范围：召回（event_log.search / episodic retrieve / RAG 注入）+ 纠偏（trait / user_identity / integrity_check）。
> 结论先行：两个 bug 不是两处独立 bug，是**同一个架构缺陷的两种表现**。建议不要单点打补丁。

---

## 〇、一句话结论

当前记忆把**"谁说的"和"什么时候发生的"这两个硬事实，当成软信息在 LLM 摘要链里反复重写**。
- "谁说的" 靠的是正文里的字符串前缀 `**用户**：` / `**叶瑄**：`，没有结构化字段 → 召回拼接时塌缩 → **bug 1（角色的话被当成用户的话引用）**。
- "什么时候发生" 用的是 `timestamp = 处理时刻`，没有"事件真实发生时间"字段 → 重新晋升/重新召回会刷新时间戳 → **bug 2（两个月前的事被渲染成"几小时前"）**。

人脑式记忆（衰减、强度、情绪质感、模糊化）本身没问题——问题是**把硬事实也一起模糊化了**。
推荐方案：**把"账本"（硬事实：谁/何时，只追加、永不重写）和"印象"（软信息：情绪/显著度/强度，允许模糊）拆开**。让人脑式的模糊只活在印象层，每条印象锚定到一条不可变的账本行；凡是要渲染"谁"或"何时"，一律回账本读，绝不从被重写过的摘要里猜。

## 一、Bug 复盘（带代码定位）

### Bug 1：角色把自己说的话当成用户说的，并引用

**现象**：`event_search`（相关往事）层里出现 `**用户**：……我没有这么写，但我确实想过；我承认我留了些余地……我放水了`，整段明显是叶瑄第一人称（下棋放水），却挂着"用户"标签。

**链路**：
1. `event_log.append()`（`core/memory/event_log.py:219`）把说话人编码成正文前缀：`role_label = "用户" if role=="user" else char_name`，再 `line = f"**{role_label}**：{content}"`。**说话人是字符串，不是字段。**
2. `event_log.search()`（`event_log.py:306`）逐行做关键词命中，命中的行**带着前缀整行**塞进 `block_hits`，再 `" ".join(block_hits)[:80]`，多个块用 `"; "` 拼接（`event_log.py:357-366`）。结果是一团：说话人前缀互相穿插、每块被砍到 80 字（产生"……"断句）、跨说话人碎片被缝在一起。
3. `prompt_builder` 层 6b 把这团字符串**原样**塞进 `<相关往事>`（`core/prompt_builder.py:751`），不带任何结构化说话人信息。
4. 额外的"投毒"来源：assistant 回复**自身**可能含字面量 `用户：…`（模型自己写对白、把两边台词都写了）。`reality_output_scrubber` 删动作描写，但不拦"用户："这种角色标签，于是它作为 assistant 块的正文落进 event_log，成了一颗定时雷。

**为什么模型会"引用"**：它在召回里读到一段挂着"用户"标签的自己的话，于是在本轮把自己过去的话当成用户说过的复述出来。

> 三个机制（写入时正文含标签 / 搜索时跨说话人缝合 / 渲染时无结构归属）指向同一病根：**说话人归属没有作为结构化元数据贯穿全链路。**

### Bug 2：两个月前的事被当成几小时前，还反复提起

**现象**：`6c_episodic`（fallback: recent high-strength）里 `【重要】几小时前，用户生日独自哭泣`。生日其实在两个月前。

**链路**：
1. episodic 的 `timestamp = time.time()`，赋值时机是**反思/晋升那一刻**（`fixation_pipeline.py:739`），不是事件发生时刻。
2. `format_for_prompt()` 的全部时间锚（刚刚/几小时前/今天/前几天/N个月前）都从 `now - timestamp` 算（`episodic_memory.py:580-593`）。
3. **过去事件没有真实发生时间**：`event_time` 只为 `temporal_ref=future` 的 hint 解析（`fixation_pipeline.py:757`），past 事件永远 `event_time=None`，时间锚只能退回用处理时刻。
4. "生日"→`first_tags`→`is_core=True`（`episodic_memory.py:223-226`）。核心记忆：不衰减（`decay_all` 跳过，`episodic_memory.py:534`）、不被自动 resolve（`_resolve_matching_open_episodes` 跳过 is_core，`fixation_pipeline.py:321`）、不参与上限裁剪。每次被召回还 `strength += 0.15`（`episodic_memory.py:475`）→ **回忆永动机**。
5. 当用户最近又提到生日（或 sweep 把旧 mid_term 重新晋升），会生成一条**全新 timestamp** 的 episode 描述两个月前的事 → 渲染成"几小时前"。
6. fallback 召回（截图里的 "recent high-strength"）**只看 7 天窗口**（`episodic_memory.py:646`）。它能命中，正是因为 timestamp 被刷新到了近期。也就是说，时间戳错误**反过来让陈旧记忆获得了优先复活权**。
7. 去重只比对最近 10 条 `narrative_summary`（`episodic_memory.py:189`）→ 同一事件滚出窗口后会作为"新"记忆反复晋升。

> 病根：**记录时间 ≠ 事件时间，且全部时间渲染用的是记录时间。** 对"回顾型"或"被重新加工"的记忆，渲染的"何时"必然失真。

## 二、纠偏机制审计（trait / identity / integrity）

用户点名要看的"纠偏"功能，逐个评估：

| 机制 | 文件 | 干什么 | 评价 |
|---|---|---|---|
| trait_tracker | `core/memory/trait_tracker.py` | 关键词命中滑窗，标记 underrepresented 特质，给 author note 加权 | 健康、低风险，但**纯词面**（`keyword in line`），且作用在"角色表达多样性"这条轴上，**和两个 bug 无关**，也救不了它们 |
| user_identity 固化 | `fixation_pipeline.py:_synthesize_identity` | counter_evidence、confidence 衰减、maturity_factor，矛盾累积不硬翻转 | 设计本身不错，是"慢真相"层。但它**完全建立在 episodic 质量之上**：上游 episode 的时间/归属错了，它只会把错误平均进 identity。纠偏在污染的下游，治不了上游。 |
| integrity_check | `core/integrity_check.py` | 确定性格式纠察（有无标题、有无中文括号/引号） | 只查**格式**，不查**语义**。没有任何一条规则会拒绝"正文里含说话人标签"或"时间戳不可能"的写入。 |

**结论**：现有纠偏全部是"下游 + 格式 + 词面"层面的修补，**没有任何一层在守护"谁说的"和"何时发生"这两个硬不变量**。这正是为什么越往人脑式（软化、再叙述、衰减）走，bug 越难根治——纠偏和病根不在同一层。

## 三、把问题拆开

**大问题（一个）**：硬事实（说话人、事件时间）没有作为结构化、不可变的元数据贯穿"turn → mid_term → episodic → identity → 召回 → 渲染"全链路；每一步 LLM 再摘要都是一次有损再叙述，可丢可换说话人、可重盖时间。

拆成可独立施工的小问题：

- **S1 说话人结构化**：event_log / short_term 的说话人必须是字段，不是 `**用户**：` 前缀。Markdown 只是"渲染产物"，不是真相源。
- **S2 召回不缝合、不截断跨说话人**：`event_log.search` 不得把不同说话人的碎片 `join` 成一段、不得砍到 80 字produce"……"。每条召回行要么带明确归属，要么转成第三人称事实（"你提过…" / "叶瑄当时说…"），让模型无从混淆。
- **S3 输出闸拦截嵌入式说话人标签**：assistant 回复里出现字面 `用户：` / `叶瑄：` 等角色标签的，写 event_log 前剥离或拒绝。
- **S4 双时间戳**：每条记忆带 `recorded_at`（入账时刻）与 `event_time`（真实发生时刻）。past 事件的 `event_time` 必须从来源 turn 的时间或正文显式日期解析，**禁止默认 now**。渲染时间锚一律读 `event_time`。
- **S5 去陈旧优先权 + 灭永动机**：fallback 用 `event_time` 而非 `recorded_at` 卡窗口；核心记忆不靠"近期高强度"复活；召回强化设冷却/上限，防 `strength` 单调爬升。
- **S6 按血缘去重**：episodic 去重用 `source_mid_ids`/血缘而非"最近 10 条文本相似"，杜绝同一事件反复"刷新晋升"。

## 四、推荐方案：账本 / 印象 分层（Option A，推荐）

核心原则一句话：**硬事实进账本（只追加、永不重写、字段化）；软信息进印象（可衰减、可模糊、可再叙述），每条印象锚一条账本行。**

```
┌────────────────────────────────────────────────────────┐
│ 账本 Ledger（硬事实，append-only，不进 LLM 再叙述）       │
│   turn_id | speaker(枚举) | event_time(真实) | text       │
│   ——event_log / short_term 的真相源；Markdown 只是渲染   │
└────────────────────────────────────────────────────────┘
              ▲ 每条印象用 turn_id / event_time 锚定
┌────────────────────────────────────────────────────────┐
│ 印象 Impression（软信息，允许人脑式模糊）                 │
│   mid_term / episodic / identity / mood                  │
│   strength / emotion_texture / 衰减 / 显著度——尽管模糊   │
│   但"谁/何时"渲染时回账本读，绝不自己编                  │
└────────────────────────────────────────────────────────┘
```

要点：
1. **说话人永不进正文**。account 行的 speaker 是枚举字段。Markdown 渲染层才拼 `**用户**：`，且渲染只用于人类导出，不用于召回喂模型。
2. **召回喂给模型的，是结构化或第三人称化的事实**，不是带标签的原始字符串拼接。这样即使印象层模糊，归属也不会错。
3. **两个时间戳**。任何要说"何时"的地方读 `event_time`；`recorded_at` 只用于运维/排序。
4. **人脑式的衰减/强度/情绪质感全部保留**——它们只活在印象层，模糊在这里是特性不是 bug。

> 这回答了你的疑问："一个劲往人脑式记忆靠，会不会有别的方案？"——有。**不是放弃人脑式，而是给人脑式套一个不可变的硬骨架。** 大脑能容忍记错时间/记错谁说的；聊天机器人把它逐字复述出来就翻车。骨架负责不翻车，血肉负责流畅。

### 备选方案（一并列出，供你拍板）

- **Option B（大改）**：retrieval 全面换 embedding + 单一结构化事件库（typed fields），砍掉多级 LLM 再叙述级联。更干净，但工作量大；而且**光上 embedding 不解决"谁/何时"**——若记录仍无结构化 speaker/event_time，向量召回照样错。所以 B 也必须先吃下账本原则。性价比不如 A。
- **Option C（只打补丁）**：就地修这两个 bug（见第五节 P0）。最省事，但归属/时间的"概念混淆"还在，会不断长出新变种。建议**至少做到 P0，并把 A 作为方向**。

**我的推荐：A。** 先按第五节 P0 止血（CC 一两天能落），同时按 P1/P2 渐进迁移到账本/印象分层，不需要推倒重来——现有各层基本可原地改造。

## 五、施工计划（分阶段，给 CC）

### P0 — 止血（直接对应两个 bug，互不依赖，可并行）

**P0-1 召回不再缝合/截断跨说话人（治 bug 1 显性面）** · `event_log.py:search`
- 命中聚合时，**按说话人分别成行**，保留行级 `role`；不要 `" ".join(block_hits)`。
- 输出改成结构化：每条返回 `{role, event_time, text}`，由 prompt_builder 渲染成无歧义的第三人称，如 `（约N天前）你说过：…` / `（约N天前）叶瑄回应：…`。
- 去掉 `[:80]` 的裸截断；要截断就整句截，并明确单说话人。

**P0-2 输出闸拦截嵌入式说话人标签（治 bug 1 投毒源）** · `core/reality_output_scrubber.py`
- 写 event_log/short_term 前，剥离/拒绝正文中出现的 `^\s*(用户|叶瑄|<char_name>)[:：]` 行。
- 加单测：assistant 回复含 `用户：…` 时不得落库。

**P0-3 时间锚改读 event_time，past 事件不准默认 now（治 bug 2 根）** · `fixation_pipeline.py:reflect_to_episodic` + `episodic_memory.py:format_for_prompt`
- episode 增 `event_time`：来源 mid_term/turn 的真实时间（取被反思批次里最早/代表性 turn 的时间），**而不是 `time.time()`**。
- `format_for_prompt` 的时间锚改用 `event_time`；`timestamp` 字段保留为 `recorded_at` 仅排序用。
- 校验：`reflect` 写入前，若 `event_time` 缺失或晚于 now，记 warning 并回退到来源 turn 时间，禁止落到"刚刚/几小时前"。

**P0-4 灭回忆永动机 + 去陈旧优先权（治 bug 2 复发）** · `episodic_memory.py`
- `retrieve` 的 `strength += 0.15` 加冷却：同一 ep 在 N 小时内重复召回不再加（或设强度上限 0.9）。
- `retrieve_fallback` 的 7 天窗口改用 `event_time`，核心记忆**不**通过"近期高强度"无条件浮起。

### P1 — 结构化骨架（账本字段落地）

**P1-1 说话人字段化** · short_term / event_log 写入层
- 内部存储增 `speaker` 枚举字段；Markdown 仅作渲染产物。迁移期：读旧 `**用户**：` 前缀兜底解析，新写入一律带字段。

**P1-2 双时间戳贯穿** · mid_term / episodic schema
- 全链路携带 `event_time`（首次入账时即固定，后续再摘要不得覆盖）。`recorded_at` 单独存。

**P1-3 按血缘去重** · `episodic_memory.write_episode`
- 去重键用 `source_mid_ids` 血缘，替代"最近 10 条文本相似"，杜绝同一事件反复晋升刷新。

### P2 — 召回质量与纠偏上移（可选，按需）

**P2-1 召回统一成"第三人称事实卡"**：episodic / event_search / mid_term 注入统一为带明确归属和明确时间的短句，模型侧彻底无从混淆谁/何时。

**P2-2 纠偏上移到入账层**：integrity_check 增语义级规则——拒绝"正文含说话人标签""event_time 不可能""past 事件 event_time=now"的写入。把守门从下游 identity 提到上游 ledger。
