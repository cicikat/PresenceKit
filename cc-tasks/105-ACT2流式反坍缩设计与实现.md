# Brief 105 · ACT-2：流式路径反坍缩（设计裁决 + 实现）

> 来源：known-issues ACT-2（open）。位置 `core/pipeline.py::run_llm_stream()`。
> 依赖：无，但建议在 102 之后做（观测数据准了再调）。

## 设计裁决（本单拍板）

两案对比：

- A 暂缓前 N token：能真丢弃重试，但引入首 token 延迟（流式的核心价值
  就是低首字延迟），且 N 取小了漏检、取大了等于半非流式，还要处理
  "暂缓窗口后才出现的重复"这个无解尾巴。
- B 流式只接受软降级：不丢弃已发内容；检测到坍缩（重复句首等既有
  信号）时记录 + 在**后续轮次**生效（如下一轮注入反坍缩提示 / 临时
  调 temperature / 标记该回复供 slow-queue 复盘）。

**采用 B。** 理由：流式内容既已可见，"用户看到又收回"比"偶尔一次
重复开头"体验伤害更大；坍缩是趋势性问题，跨轮纠正足够，不必单轮内救。
A 案封存，除非上线后出现单轮坍缩率显著高于非流式的数据再复议。

## 实现

1. 流式完成后跑与非流式同一套坍缩检测（复用现有检测函数，勿复制）。
2. 命中时：打观测日志（含 uid/char_id/命中规则）；写入下一轮可读的
   降级信号（存放位置遵循 sandbox.get_paths()，勿硬编码）。
3. 下一轮 build_prompt 读到信号时注入反坍缩提示层（带 `_layer` 字段
   + `_drop_priority`，遵守 Hard Rules 3）。信号一次性消费。
4. 测试：流式坍缩样本 → 断言不中断输出、信号写入、下一轮提示注入、
   再下一轮信号已清除。
