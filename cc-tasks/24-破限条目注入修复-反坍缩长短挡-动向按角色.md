## 1. 破限条目多选只注入第一条（bug 修复，优先级最高）

### 1.1 根因（已确认）

系统里存在**两套并行的破限存储，其中一套从未被注入**：

| 存储 | 写入方 | 读取方 |
|---|---|---|
| `characters/reality/jailbreaks/{stem}.json`（base/anti_assistant/style/intimacy/memory_identity） | 手工/资产管理 | `core/prompt_builder.py:218 _load_jailbreak()`，受 `active_prompt_assets.json` 的 `enabled_jailbreaks` 控制，注入层 0/2/11 |
| `characters/reality/jailbreak_entries.json` | 前端「偏好→世界→破限条目」EntryManager → `admin/routers/jailbreak_entries.py` CRUD | **全仓无任何 prompt 路径读取此文件**（grep 确认只有 router 自己读写） |

`prompt_builder.py:350` 的注释声称层 0 来自 `jailbreak_entries.json`——注释是错的，代码读的是 stems。

用户看到「只有第一条生效」的假象：条目「情绪张力」恰好同时存在于 `base.json`（enabled，layer 11）里，而 `base` 在 `enabled_jailbreaks` 中；其余勾选条目（性张力/睡觉/1）只存在于 `jailbreak_entries.json` 和**未启用**的 `intimacy.json` 里，所以永远不注入。

### 1.2 修复方案

改 `_load_jailbreak(layer)`（`core/prompt_builder.py:218`）：

1. 现有 stems 逻辑不动。
2. stems 之后追加读取 `get_paths().jailbreak_entries()`，同样按 `enabled` + `layer` 过滤，追加到 `parts`。
3. **按内容去重**：`content.strip()` 相同的条目只注入一次（「情绪张力」在 base.json 和 entries.json 中重复存在，必须去重）。
4. 修正 350 行注释；`jailbreak_entries.json` 读取失败时 fail-open（沿用现有 try/except 风格）。

### 1.3 测试与验收

- 新增测试（可放 `tests/test_prompt_layers.py` 或新文件）：entries.json 多条 enabled、不同 layer → 各自注入到 0/2/11；与 stem 文件内容重复 → 只出现一次；disabled 条目不注入。
- 手工验收：前端勾选多条破限后发一条消息，用 prompt 检视器确认全部出现。
- 文档同步：`docs/prompt-layers.md` 破限层说明补上双来源+去重规则。

## 2. 反坍缩重构（「嗯。」坍缩治理 + 字数挡位简化）

### 2.0 现状盘点

| 机制 | 位置 | 逻辑 |
|---|---|---|
| S2 防句式坍缩 | `core/memory/short_term.py:462 detect_reply_homogeneity` → `prompt_builder.py:1160` 注入 author_note | 近 6 条 assistant 前 2 字相同 ≥3 次 → 软提示（文案里**复读了字面前缀**） |
| 9_anti_repeat 层 | `prompt_builder.py:264 _recent_openings` + ~1093 | 近 3 条前 8 字列出，要求换开头（同样复读字面） |
| S3 防字数坍缩 | `short_term.py:495 detect_reply_length_collapse` → `prompt_builder.py:1169` | thresholds (15,40,80,150) 分 5 挡，近 5 条同挡 → 通用软提示 |

### 2.1 问题 8：字数挡位简化为长/短 + 非对称触发

设计意图（用户确认）：**短句难触发、长句易触发**——模型爱往注水长句坍缩，短句反而更像活人；5 挡太密会限制模型。

新配置（替换 `anti_collapse.thresholds` / `recent_n`）：

```yaml
anti_collapse:
  enabled: true
  short_max: 60        # 字符数 < 60 为短句，>= 60 为长句（单边界，只分两挡）
  recent_n_long: 4     # 近 4 条全为长句 → 触发（易）
  recent_n_short: 7    # 近 7 条全为短句 → 才触发（难）
```

改 `detect_reply_length_collapse`：

- 签名改为 `(history, *, short_max=60, recent_n_long=4, recent_n_short=7)`。
- 先看近 `recent_n_long` 条是否全为长句 → 命中返回**长句版提示**：
  `（最近几条都挺长，这次收短——去掉铺垫和水词，捡最有劲的一两句说。）`
- 否则看近 `recent_n_short` 条是否全为短句 → 命中返回现有的通用打破惯性文案。
- 调用点 `prompt_builder.py:1174-1180` 同步改参数读取；`config.yaml` / `config.example.yaml` 同步（旧 `thresholds`/`recent_n` 键读到时忽略并打一条 warning，不做迁移）。
- `tests/test_anti_collapse.py` 按新逻辑重写。

### 2.2 问题 7：软提示已触发但「嗯。」坍缩仍持续 —— 治理 + 止血

根因分析（写进代码注释）：

- 自我模仿强于纠偏：层 9 历史里模型看到自己近 5-6 条全以「嗯。」开头，一条 author_note 软提示对抗不了整段上下文示范。
- 现有提示文案**复读了字面前缀**（「连续用了『嗯』…」），反而再次 prime 这个 token。

三层措施，按顺序落地：

**(a) 历史投影去同质（主修，止血+治本）**
`build()` 组装层 9 历史时：若 S2 检测命中前缀 P，且 P 属于填充词白名单（`嗯 啊 呃 哦 唔 哈` + 可跟 `。，…～`），则对注入 prompt 的 assistant 历史**副本**做处理——保留最早一条的开头 P，其余各条剥掉开头的 P。效果：模型不再从上下文学到「每句都嗯。」。
⚠️ 硬约束：只改 prompt 投影，**绝不写回 short_term 存储**；动手前先读 `_sanitize_assistant_message()`（`core/memory/short_term.py:118`）确认与其无叠加副作用（本改动在 build 侧、sanitize 在写入侧，理论无交集，但需确认历史内容此时已是脱敏后文本）。

**(b) 提示文案避免 priming**
S2 命中填充词前缀时，文案改为不复读字面：
`（你最近几条开头都是同一个语气词，这次第一个字直接进正文——从动作、称呼或要说的事本身开始。）`
非填充词前缀（如「现在，」）保留现有引用式文案。

**(c) 输出端校验重试（硬止血，可配置）**
位置：`core/pipeline.py` `run_llm()` 之后或 `core/llm_output_validator.py`（选后者更顺，已有失败计数设施）。
逻辑：pipeline 侧在 build 前用同一 `detect_reply_homogeneity` 检测一次得到 P；若输出仍以 P 开头 → 追加一条强 system 指令重试 1 次；重试仍命中且 P 为填充词 → 剥掉开头 P 后接受。
配置：`anti_collapse.prefix_retry: true`（默认开）。注意重试计入 LLM 调用成本，日志打 `[anti_collapse] prefix retry`。

### 2.3 验收

- 构造 6 条「嗯。xxx」历史 → 检查注入的 messages 中历史副本只剩一条带「嗯。」、author_note 出现新文案。
- 单测覆盖 (a) 的白名单/非白名单分支、(c) 的重试与剥离分支。
- `pytest` 全绿；本次不改 `tag_rules.py`，无需跑 `run_eval.py`。

## 3. 动向（activity）按角色隔离（问题 1 的后端半）

### 3.1 根因（已确认）

- `core/activity_manager.py` 全局单状态：`_load_state()/_save_state()/_load_pool()` 调 `get_paths().activity_state()/activity_pool()` 时**不传 char_id**，全部落在默认角色（yexuan）路径上。
- `admin/routers/activity.py` `GET /activity/current` 的随机池动向因此永远是 yexuan 的，与 `active_character` 无关（梦境/共同活动分支已按 active char 取，没问题）。
- 对照：`admin/routers/mood.py` 已有 `_active_char_id()` 正确做法；`admin/routers/reading.py` 全程 `_active_char_id()`，阅读会话存储本身是隔离的。

### 3.2 改动

1. 把 `mood.py` / `reading.py` 里重复的 `_active_char_id()` 抽成公共 helper（建议 `admin/routers/_common.py` 或 `core/asset_registry` 侧），三处复用。
2. `activity_manager` 全链路加 `char_id` 参数：`get_current(char_id)` / `should_switch` / `switch_activity` / `_load_state` / `_save_state` / `_load_pool` / `get_prompt_fragment`。grep 全部调用点同步（至少：`admin/routers/activity.py`、prompt 注入处、scheduler 触发器）。prompt 注入侧用当前 pipeline 的 char_id，路由侧用 `_active_char_id()`。
3. 角色池 fallback：`_load_pool(char_id)` 若该角色的 `activity_pool.yaml` 不存在 → fallback 读默认角色池（不复制文件），打 debug log。
4. `GET /activity/current` 响应体**新增 `char_id` 字段**——前端要用它做动向时间轴的按角色隔离（见配对 Brief 14 §B）。
5. `_TRANSITION_CHARACTER_INNER` 兼容写（`_save_state` 里写旧路径那段）只对默认角色保留。

### 3.3 顺手核查（不强求修复，修不动就记 known-issues）

用户报告「阅读动向被记录进 yexuan 数据」。`reading.py` 路由侧已隔离，怀疑点：(a) 前端动向时间轴是全局 localStorage（前端 Brief 处理）；(b) 阅读伴聊/summary 落主记忆时 char_id 是否有缺省——检查 `core/activity/reading_companion.py` / `activity_summary.py` / `core/activity/transcript.py` 里所有 `char_id` 默认值调用。发现缺省即补，写 `docs/known-issues.md`。

### 3.4 验收

- 切 active_character 后 `GET /activity/current` 返回该角色自己的随机池动向 + `char_id`。
- 阅读/五子棋/国际象棋分支行为不变。
- 文档同步：`docs/scheduler.md` 或 `docs/channels.md` 中动向相关描述、`AGENTS.md` 关键文件表如有涉及则更新。
