## A 组：Bug 修复（可立即施工）

### 工单 A1 — 群聊梦境卡死排查与修复

**现象**：
- 用户发第一句话无任何反应；发第二句话后才出现"…"待机气泡。
- 有时卡死在"…"状态，后端无任何 LLM 调用记录（纯卡住，非慢）。

**定位入口**：`core/stage/dream_runtime.py`（`run_dream_stage_turn()`）、`core/stage/dream_state.py` / `dream_store.py`、端点 `admin/routers/group_dream.py`；先读 `docs/stage.md` §六。

**排查方向（按可能性排序）**：
1. **状态锁/session 状态未释放**：第一句话被吞、第二句才触发，典型症状是首条消息进入时 session 处于某个中间态（如上一轮未正常收尾），消息被静默丢弃或排队后无人消费。检查 dream session 的轮转状态机是否存在"收到消息但不调度回合"的分支，尤其是异常路径（LLM 超时/校验失败）后状态是否回滚。
2. **"…"卡死且无后端调用**：说明前端收到了 typing/待机信号但后端回合从未启动或在 await 处悬挂。检查 `run_dream_stage_turn()` 内是否有无超时的 await（锁获取、队列 get、WS send）。
3. 修复必须包含：**全链路超时兜底**（回合级 timeout，超时后重置状态并给前端明确失败信号，而非永久"…"）+ **状态机日志**（进入/离开每个状态打一条 debug 日志，方便下次复现时定位）。

**验收**：能构造复现场景的回归测试（模拟 LLM 悬挂 → 回合超时 → 状态复位 → 下一条消息正常处理）；`pytest --testmon` 通过。

### 工单 A2 — 群聊无响应排查 + 日志降噪

**现象**：
- 已配置"每轮必须 1 人回复"，但有时说话无人响应；此时后端有角色加载日志和带 DeepSeek URL 的请求日志。
- 该 URL 日志过多像噪音，且有时会莫名连刷多次。

**定位入口**：`core/stage/arbiter.py`（回合仲裁）、`core/stage/runner.py`；日志侧 `core/llm_client.py`。

**两个子任务**：
1. **仲裁修复**：检查"每轮必须 1 人回复"约束在什么路径下会被绕过——重点看：候选角色全部被闸门（gating / speak 闸）拦下时是否有兜底强制发言；LLM 返回空/校验失败后是否重试或静默放弃。日志显示"角色加载 + 发出了 LLM 请求但无产出"，怀疑是 LLM 返回被 `llm_output_validator` 判失败后直接吞掉。失败时必须降级兜底（重试一次或强制指定发言人）。
2. **日志降噪**：
   - LLM 请求 URL 日志降为 `DEBUG` 级；`INFO` 级只保留一条精简的"对话 API 调用"日志（模型名 + 用途 + 耗时），不含完整 URL。
   - "连刷几次"的情况：先确认是重试机制（`run_llm` retry）导致的正常重试日志，还是真的重复调用。若是重试，日志加上 `retry 2/3` 标注；若是重复调用，那本身就是 bug，单独修。
   - 顺带梳理 `core/llm_client.py` 的日志层级规范，写进代码注释。

**验收**：正常一轮群聊 INFO 日志 ≤ 每次调用 1 行；构造"全部角色被闸"场景测试兜底发言生效。

### 工单 A3 — 视觉观测无数据排查

**现象**：已在配置里填了 GLM（`vision.provider: glm` / `glm-4v-flash`）并启用了功能，但观测里无数据。

**定位入口**：`core/perception/vlm_client.py`、`core/coplay/observer.py`、`core/media_processor.py`、config 的 `vision:` 块。

**排查步骤**：
1. 确认配置读取链路：config → vlm_client 初始化是否真的拿到了 key/model（打印初始化日志确认）。
2. 确认触发链路：视觉观测的调用点在哪（media_processor 收图？coplay observer 截屏？），是否有开关默认关闭、或触发条件从未满足。
3. 确认失败是否被静默吞掉：VLM 调用异常时是否 fail-open 无日志。所有失败路径补 `WARNING` 日志。
4. 修好后，观测数据的落点必须有只读观测端点（强制规则 7）。

**验收**：手动发一张图/触发一次观测，观测端点能看到数据；失败时日志可见原因。

## B 组：功能新增

### 工单 B1 — 探针工具：角色自改记忆（episodic + 用户画像）

**需求**：用户说"你记错了"之类时，角色能自己修改自己的 episodic 记忆和用户画像。**设计决策（已拍板）**：不给用户提供手动查看/删除/修改入口——修改由角色自主完成；但修改行为本身必须可见（工具执行结果回显给对话 + 留痕）。

**实现要点**：
1. 新工具建议两个：`revise_memory`（改/弱化某条 episodic）、`revise_user_profile`（改用户画像字段）。注册进 `_TOOL_REGISTRY`，补 `examples` + `keywords`（探针 prompt 自动同步）。归入 `memory` 类目（生成侧 tool loop 默认已暴露 memory）。是否同时暴露给 pre-pipeline 探针需评估：建议**只走生成侧**，因为"记错了"的判断需要上下文，探针太早。
2. episodic 修改语义：优先"降强度 + 追加更正"而非物理删除（与遗忘=降级的既有设计一致，见 `docs/memory.md`）；用户画像（`data/user_identity/{uid}.yaml` 层）允许字段级覆写。
3. **强制留痕**：每次修改调用 `provenance_log.append()`（强制规则 6），并写 `action_trace`（Brief 27 机制），使"角色改了自己的记忆"出现在跨轮动作痕迹层。
4. 工具结果文本要让角色能自然转述（"我把那段记忆改过来了"），同时管理面板观测端点可查修改历史（强制规则 7）。

**前置阅读**：`docs/tools.md`、`docs/memory.md`。
**验收**：对话中说"你记错了 XX"→ 探针/loop 触发工具 → 文件实际变更 + provenance 有记录 + 观测端点可查。

### 工单 B3 — AA更新.bat 加固 + release 包兼容

**结论先行**：`AA更新.bat` 是**有用且安全**的——它只做 `git pull` + 依赖同步（uv pip sync 或 pip install），**不会覆盖 data/、config.yaml、secrets**（git pull 不动未跟踪/本地修改文件；依赖同步只动 .venv）。用户此前不敢用的顾虑可以打消，但值得加防手抖：

1. **运行前检查**：检测 `python main.py` 进程是否在跑（tasklist 查 .venv python），在跑则提示"请先停止服务再更新"并退出。
2. **本地改动保护**：`git status --porcelain` 非空时列出改动文件并要求确认（防止 pull 冲突把人吓到）；pull 失败（冲突）时给出中文提示而非裸报错。
3. **release 包兼容**：可以做到。方案：脚本开头检测 `.git` 目录——存在走 git pull 路径；不存在（release zip 用户）则提示"当前为发行版，请下载新版 zip 覆盖程序目录（data/ 与 config.yaml 不要覆盖）"，或进一步做成从 GitHub Releases API 拉最新包、解压覆盖白名单目录（`core/ admin/ channels/ main.py` 等代码目录）、跳过 `data/ config.yaml secrets*`。第一版先做提示即可，自动更新可留待后续。
4. 更新后打印当前版本（git describe 或 VERSION 文件）。

**前置阅读**：`docs/release-guide.md`。

### 工单 B4 — 思考笔记文风修复

**现象**：`data/very_formal_project/思考笔记.txt` 是给角色自由书写的文件，但内容被写成了生硬的记录体。

**根因（已定位）**：`core/post_process/toy_autogrow.py::_judge_turn()` 的 prompt 问题：
- 指令是"输出一句话（不超过50字）**描述你想记住什么**"——这是摘要指令，不是日记指令，模型自然产出记录体。
- 用的是 `summary` 路由模型 + temperature 0.3，进一步压向干瘪。

**修复方案**：
1. Prompt 重写为第一人称日记语气，例如："以{char_name}的口吻，像随手写日记一样记下此刻的想法或感受，可以有情绪、比喻、碎碎念，1~3 句"；判断"是否值得写"与"写什么"可拆成两步或保留单步但把 SKIP 判据放前面。
2. temperature 提到 0.9 左右；模型路由考虑从 `summary` 换成 `chat`（人格模型），token 成本略升但该功能低频（有冷却时间），可接受。
3. 可选：prompt 里给 1~2 条好例子（few-shot）锚定文风。

**验收**：跑几轮对话后人工检查新增条目文风；不需要自动化测试（主观质量），但改动本身跑 `pytest --testmon`。

### 工单 B5 — 管理面板：API 调用总账观测

**需求**：管理面板观测区新增"调用记录"页——任何对外 API 调用（对话 LLM、summary、VLM、TTS、web search、embedding……）都记一条：**调用方是谁（哪个模块/功能）、用途、模型/服务、耗时、产物指向（写入了哪个文件/层）**，并按种类分组展示，解决"忘记到底有多少地方在用"的问题。

**实现要点**：
1. 在 `core/llm_client.py` / `core/model_registry.py` 的统一出口处埋点（所有 LLM 调用都过这里，天然收口）；VLM/TTS/web search 各自出口补同款埋点。埋点写入统一的 `api_call_log`（jsonl 滚动文件，路径走 `sandbox.get_paths()`，按天滚动、保留 N 天）。
2. 记录字段：`ts / caller（调用点标识，埋点处显式传入） / purpose / provider / model / duration_ms / ok / output_hint（产物落点，可空）`。caller 标识顺便就是一份"全系统调用点清单"。
3. 面板端点：`GET /observability/api-calls`（分页 + 按 caller/provider 过滤 + 按种类聚合计数）。scope 按观测类选取（见 `docs/security.md`）。
4. 埋点必须 fail-open（记录失败不影响调用本身）。

**协同**：A2 的日志降噪与本工单互补——INFO 日志变少，详细信息进总账。建议 A2 先行或同批施工。

### D2 — 人机直连模式：模型路由 / 探针裁剪 / MCP vs 自写工具（对应第 8 条）

用户的思路（此处记录，供讨论）：
- 纯人机（非角色扮演）+ 高级模型（Claude/GPT 类）时，探针系统只是徒增 token——因为架构里约六成设计是为了让不太聪明的模型跑起来。
