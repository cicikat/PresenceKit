## 二、工单写作约定（必须遵守，用户已习惯此格式）

1. 格式：标题 `# Brief N · 名称`，头部引用块写依赖/参考/现状问题，正文
   `改法（带文件路径）→ 拍板 → 测试 → 不做什么` 四段。参考范本：`cc-tasks/34`、`47`。
2. **拍板文化**：能拍死的初值直接拍（命名常量），不留开放问题给 CC；只有不可逆决策
   （删数据/改契约/花真钱）才升级给用户。
3. 一切新功能**默认关**（config 开关），一切新落盘走 `get_paths()`/data_paths 访问器
   （Hard Rule 1），新记忆写入点必带 `provenance_log.append`（Hard Rule 6），
   文案用 `char_name`/`user_name` 插值（Hard Rule 8）。测试指令一律 `pytest -n auto`。
4. 交付时标注波次（可并行/串行/依赖），用户会整批丢给 Claude Code 执行。
5. 用户偏好：中文；语气可以轻松（她用 🐱/咪，可轻度回应但别过量）；讨论先行、
   工单后行——她说"先讨论"就别急着写文件。

## 三、62（习惯固化）的调整指南

准入条件在工单头部。观察数据不达标时**不要硬跑**，按症状调：

- **幻觉率 ≥ 10%**：先调 56 的 VLM prompt（枚举收紧、confidence 自报校准）或换更强
  本地模型，重新累计 14 天；不要靠"聚合会稀释幻觉"自我安慰——8 天阈值挡不住系统性幻觉
  （模型把"看网页"一律标成"working"这类偏置会直接固化成假习惯）。
- **sensitive 漏判 > 0**：这是硬红线，修 56 的敏感判定（客户端窗口黑名单 + VLM 双闸）
  后归零重测，期间 62 冻结。
- **分布与作息不符**：多半是采样偏置（只在某几个时段抓帧），修抓帧端节律而非后端。
- 达标后若用户要求更细的习惯粒度（如区分工作日/周末），在聚合规则里加 day-of-week
  维度即可，勿加 LLM 调用。

## 四、63（受限购物）的调整指南与坑图

准入三条见工单头部，第 3 条（用户显式拍板）不可代拍。已知坑（讨论中反复确认过）：

1. **提示注入 × 金钱**是最坏组合——工单的三段隔离（选品脏/确认鉴权/执行无浏览器）
   是核心设计，任何"简化"提案（比如让执行段顺手抓个页面确认库存）都要拒绝。
2. **国内支付现实**：支付宝/微信无 agent API，扫码主流程 = 风控雷区（封号）。工单已
   把"无 API 商家不支持"定为 v1 边界；如用户想扩，方向是平台开放接口（如电商平台的
   免密小额/礼品卡兑换），不是浏览器自动化。
3. 聊天内自然语言确认（"嗯买吧"）v1 明确不做——确认必须是带鉴权的显式动作，
   防注入伪造。用户如果嫌麻烦，可做的妥协是管理面板一键确认（client 面板工单已含）。
4. 法律责任主体永远是用户本人，额度 = 可接受损失上限，这是给她的提醒不是给代码的。

## 五、关键设计原则（四系列共同的世界观，续写新工单时对齐）

1. **接入 ≠ 会用**（capability vs competence）：MCP 给系统能力，成长系统管角色熟练度；
   媒介类工具必须先门控后接入，**能力只能单调递增**（收回=可感降智=叙事灾难）。
2. **慢即是叙事**：练习/成长的速率限制既是成本闸也是节拍器；三个月学会写诗比三天动人。
3. **隔离先例链**：web_echo → dream_echo → 练习产物 → 视觉 caption，任何"生成物/外部
   内容"都不得直接进 fixation 固化链，只有"事实性的一行"（action_trace）可回流。
4. **fail-closed 写入，fail-open 读取**：Write Envelope 是命根子；唯一反例是 57 台账
   （记不了账就不许花钱，写失败中止动作）。
5. **风险 = 自主度 × 访问权**：放开新动作类型时先收紧其额度与范围；信任靠审计数据
   挣，不靠配置给。
6. **一切可观测**：新落盘必带只读端点（64 已把这条写进 AGENTS.md 门禁）；改行为前
   先有 trace（44 之于记忆、50 之于仲裁、shadow 之于视觉）。
