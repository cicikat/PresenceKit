# Brief 120 · 工具循环二次调用兜底（尾部花括号方案）

写于 2026-07-24。设计讨论定稿，未实现。承接的问题：钓鱼/海龟汤这类需要模型在读到
第一次工具结果后，自己再发起一次依赖性工具调用的小游戏，目前做不到——不是"完全没有
多轮机制"，是已有的多轮机制在这一步会把模型的输出直接丢弃。下面先说清楚"已经工作的
部分"，再说清楚"确认损坏的那一点"，最后是方案。

## 1. 现状核实：Path C 本来就支持多步，问题出在最后一步的丢弃逻辑

`core/pipeline.py::run_agentic_loop()`（第 892 行起）本身是一个 `max_steps`（默认 5）
的循环：每步 `chat_turn(loop_msgs, tools, ...)` → 若返回 `tool_calls` 就执行、把结果
按 `role: tool` 塞回 `loop_msgs`、进入下一步循环。**这一步本身就允许模型在看到第一次
工具结果后再发起第二次结构化调用**——机制层面没有"只能一轮"的限制。

真正的断点在第 975-1040 行：当某一步 `chat_turn()` 返回**没有 tool_calls 但有 content
文本**时，代码判定为 `outcome = ("natural", turn.content)`；如果这是循环中**用过工具
之后**的一步（`used_tool == True`），代码会执行到第 1036-1040 行：

```python
loop_msgs.append({"role": "system", "content": _voice_reanchor(char_id)})
...
return await self.run_llm(loop_msgs, is_proactive=is_proactive)
```

也就是说：**这一步模型真正吐出来的 `turn.content`（`text` 变量）被完全丢弃，从未使用**
——`voice_reanchor` 只是往 `loop_msgs` 追加一条 system 提示，然后重新发起一次不带
`tools` 的全新生成来产出"最终回复"。如果模型在这一步试图用自然语言描述它想做的下一步
动作（比如"我抛竿等一会"），这段话会被原样吞掉，替换成一段完全不知道这个意图的重新
生成文本。

**这精确对应茶茶描述的 bug**："聊天模型给的指令变成了输出的文字，而探针又没法接收到
这个"——不是探针接不到（探针 Path A 在 tool_loop 激活时根本不会跑，见下方澄清），而是
run_agentic_loop 自己在这一点把文字吞了，无路可去。

## 2. 方案：尾部花括号，作为这一个丢弃点的救援钩子

不新增第四条工具分派路径，只在上面确认的丢弃点插入一个救援分支。约定：

- 模型在自然语言回复的**末尾**可以附加 `{true：我需要xxx功能，参数是yyy}` 或
  `{false}`/不写（默认视为 false）。
- `true` 时，方括号内的自然语言意图文本被送去解析执行；`false` 或缺失时静默忽略。
- 花括号本身 + 内部文本，在展示给用户前必须被剥离。

### 2.1 宽容解析（这是相对结构化 function-call 最大的优势，必须保留）

不做严格 JSON/格式校验。解析规则：在文本中找 `{true` 这个前缀（大小写、全半角冒号都
兼容），只要出现就认定"要调用"；截取内容到下一个 `}`，找不到闭合括号就直接截到字符串
结尾。哪怕模型漏了一个符号，顶多是截取的意图文本多带了点噪音，不会导致"整段解析失败、
啥都执行不了"。这一条要写进实现的 docstring 里，防止后面有人"顺手"改成严格校验。

剥离展示文本同理：用同一个宽容规则找到 `{true`/`{false` 起始位置，从那里截断到用户
可见文本为止（宁可切多一点，不可放过一点露出的花括号）。

### 2.2 执行方：复用探针范式，不是新写一个调度器

`core/tool_dispatcher.py::get_probe_prompt()` 已经是"给一段自然语言、判断该调哪个
工具"这件事的现成实现（目前只喂 info/desktop 两类）。这次不同的是：

- 输入不是整条回复模糊猜测"这是不是个请求"，而是模型自己已经用 `true` 显式标注过的
  一小段干净意图文本——比探针现在处理用户原话还简单，不需要再判断"这算不算意图"，
  只需要"这段话对应哪个工具、什么参数"。
- 类目应该覆盖 run_agentic_loop 当前实际暴露给这一轮的 categories（`tool_loop.categories`
  减去 `exclude_tools`，包含 mcp），而不是探针默认的 info/desktop 两类——否则钓鱼/
  海龟汤这类 mcp 工具还是够不着。

建议新增一个函数，形态类似 `get_probe_prompt()` 但读取的类目参数化（不要在
`get_probe_prompt` 里加 if 分支，另写一个共享内部 helper，两者都调它，避免探针本身
的 prompt 措辞被这次改动污染）。

### 2.3 插入点：就是刚才确认的丢弃分支，不是新状态机

在 `_run_steps()` 第 975-992 行，`turn.tool_calls` 为空但 `turn.content` 非空这一支，
先跑一次宽容解析：

- 命中 `{true: ...}` → 把提取出的意图文本交给 2.2 的解析函数，拿到工具名+参数后按
  现有 `_execute(...)` 执行、结果按 `role: tool` 塞回 `loop_msgs`，**不 return，让
  `for _step in range(max_steps)` 循环自然进入下一轮**（复用现有多步机制，不用另开
  一层循环）。展示给用户的部分（花括号剥离后的纯文本）如果非空，可以先存一份，等
  循环真正收尾（下一轮 natural 或 exhausted）时再决定要不要用它替换 voice_reanchor
  的重新生成——这一点留给实现时再定，不是这次要拍板的核心。
- 未命中 → 原有逻辑不变（`outcome = ("natural", turn.content)`）。

## 3. 顺带定名：不要再统称"探针"

讨论中发现"探针"这个词在口语里被当成了整套工具调度系统的总称，但代码里"探针"specifically
指 Path A（`get_probe_prompt` + main.py 步骤 3 的那次轻量单轮分类）。三条并行路径的
准确说法：

| 路径 | 触发时机 | 覆盖类目 | 本质 |
|---|---|---|---|
| **探针**（Path A） | `tool_loop_active(uid)` 为 **False** 时才跑（群聊/非 owner/非 function_calling 预设）；owner 私聊+function_calling 时**探针整体被跳过**（main.py 第 440-449 行 `_loop_active` 分支），决策权完全交给工具循环 | info / desktop | 单轮、一次性分类器 |
| **意图解析**（Path B） | 生成之后，`config.intent_reflex.enabled` 开且若干守卫通过 | 固定几个桌面动作（minimize/play_song 等） | 从模型自己回复的文字里抠意图，事后执行 |
| **工具循环**（Path C） | `tool_loop_active(uid)` 为 True（owner 私聊 + function_calling 预设 + 开关未被角色卡关闭） | `tool_loop.categories`（默认 info/desktop/memory，config 已加 mcp）减 exclude_tools | 主模型自己带着完整 tools schema 多步决策，目前茶茶的 27x/24x 日常对话走的就是这条 |

即：**日常和叶瑄单聊时，探针根本不会触发**，一直在跑的是工具循环。这也是为什么这次
钓鱼/海龟汤问题的真正断点在 run_agentic_loop 里，不在探针的 prompt 或类目配置里。
建议以后口语里分开叫"探针 / 意图解析 / 工具循环"三个名字，不再笼统说"探针系统"；
如果需要一个总称，可以叫"工具调度体系"，三条路径是它下面并列的三个具体机制。
