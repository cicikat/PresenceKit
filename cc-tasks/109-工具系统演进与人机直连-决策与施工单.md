## 一、决策依据（外部佐证）

2025–2026 的工程实践共识：工具选择准确率随暴露工具数**非线性衰减**——约 20 个以内是安全区，50 个时主流模型还有 84–95%，200 个掉到 41–83%，且列表中段的工具比首尾更容易被漏选；主流解法是**按场景动态暴露子集**（分层/检索式选择），Tool RAG（向量检索工具注册表）是工具上百之后才需要的重武器。

结论映射到本项目：现有 `_TOOL_REGISTRY` 全量也就二十来个，**分类子集暴露机制（`tool_categories`）已经是正确架构**，不需要 Tool RAG；要做的是把暴露面控制做成一等公民并守住"单次暴露 ≤ 20"的红线。

## 二、拍板结果

### 决策 1 — MCP vs 自写工具：自写为主，MCP 保留为可选外挂
`docs/tools.md` 的现有立场（MCP 只接工具、只经 Path C、角色卡不声明 `mcp` 类就不可见）**维持不变**，这已经是正确答案。不投入精力扩 MCP 生态；新能力一律自写进 `_TOOL_REGISTRY`。理由：自写可控、可与记忆/痕迹/闸门体系深度集成；MCP 的价值只在"接现成外部生态"，按需再开。

### 决策 2 — 不做双命名/术语预设层
原设想"给 Claude/GPT 按其训练术语做工具名预设"**不做**。现有工具名已是英文 snake_case 标准风格（`web_search` / `get_time`），与各家 function calling 训练分布天然对齐；双命名会让 action_trace、探针、文档三处产生别名映射负担，收益不成比例。改为轻量替代：**过一遍所有工具的 `description` 与 parameters 描述**，按 function calling 最佳实践重写（动词开头、明确何时该用/不该用、参数含枚举与示例），一次施工惠及所有 FC 模型。

### 决策 3 — 人机直连模式：做成 preset 组合，不新建第四条路径
架构盘点后发现基础设施已齐 90%：Path C 激活时探针（Path A）与意图解析（Path B）本来就整体跳过；`presence_ext` 已有 `disabled_layers` / `model_routing` / `tool_categories` / `proactive` 四个钩子。所以"人机直连"不是新系统，是**一张角色卡 + 一个补丁**：

- 角色卡（新增 `examples/assistant.example.json`）：`disabled_layers` 关掉角色扮演类层（jailbreak 层、author_note 轮换、mood 软提示、presence 层——具体清单施工时对照 `docs/prompt-layers.md` 逐层判断"是否服务于人格扮演"）；`model_routing` 指向 claude/gpt preset；`tool_categories: ["info", "desktop", "memory", "fs"]`；`proactive: "off"`。
- **补丁：`presence_ext.tool_loop` 钩子**。当前 Path C 激活由全局 `tool_loop.enabled` 控制，人机卡需要"这张卡激活即开 loop"而角色扮演卡维持关。新增 per-char 覆写：`presence_ext.tool_loop: "on" | "off" | 缺省(跟全局)`，消费点 `tool_dispatcher.tool_loop_active()`。FC preset 校验保留（xml_fallback 模型仍不激活）。
- **记忆系统原封不动**（用户已拍板）：fetch_context / 固化链 / provenance 全部照旧。总结概括类照旧走模型路由自选。
- 探针去除即由 Path C 激活自动达成，无需单独开关。

### 决策 4 — 暴露面红线与守卫
新增守卫测试：`run_agentic_loop()` 单次组装的 tools schema 数量 > 20 时 log WARNING（不 fail，先观测）。将来工具增多需要分场景细分时，扩展 `tool_categories` 的粒度（如拆 `desktop` 为 `desktop_media`/`desktop_window`），而不是引入检索层。

### 决策 5 — 表情包复用（D1）
已查明：表情包在 `core/output/sticker.py`，是**输出侧后处理**，不走探针、不占工具位，与工具层顺序无关。复用到两个前端 = 通道层工单：确认 `turn_sink → channels.registry` 广播的 payload 里表情包字段对 desktop / mobile 通道均下发（QQ 现有行为不动），两个前端各自渲染。这是**小工单，可直接交 cc**，不需要讨论。前端渲染部分（PresenceKit-desktop / mobile）在客户端仓库另开单。
