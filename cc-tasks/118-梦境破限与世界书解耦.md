# Brief 118 · 梦境破限（jailbreak）与世界书解耦

写于 2026-07-24。茶茶反馈：当前 sandbox 模式下，梦境的"破限"内容只能跟着世界书
一起填，但破限和世界书其实是两件事——破限可以搭配一本世界书，也可以独立存在
不挂任何世界书。

## 已核实的现状（未深挖到底，够写工单但别当结论）

`admin/static/index.html` 的"梦境设定"页（`page-dream-settings` → sandbox 创作栏）
其实**已经**把"世界书条目"（`dream-lore-card`）和"破限预设"
（`dream-preset-card`，`textarea#dream-preset-text`，保存走 `saveDreamPreset()`）
做成了两张视觉上独立的卡片，不是同一个输入框——这点和茶茶的描述（"混在世界书里
填"）不完全一致，需要先核实清楚到底是哪个环节耦合的：

- 大概率的耦合点：这两张卡片都挂在同一个"选择世界"（`dream-world-select`）
  下面，`dream-preset-card` 的显示/保存可能是**跟着某个"世界"走的一个附属字段**，
  而不是独立于"世界/世界书"之外的资产——也就是说，想写一条破限，仍然要先建一个
  "世界"容器，即使这个世界压根不需要世界书内容。对照现实侧（`characters/reality/
  jailbreaks/`，`asset_registry.py` 里 `reality_jailbreak` 是完全独立的资产类
  型，创建时不需要绑定任何世界书）——梦境这边没有对等的独立资产类型，`dream_preset`
  这个 kind 目前看起来就是"破限预设"本身（见 `_DREAM_PRESET_ID_MAP` 那几个中文
  文件名：审讯/多p/触手巢穴/感官），但走的读取路径是不是必须挂在"世界"选择下，
  需要接手的人对着 `saveDreamPreset()` / 对应后端路由和 `core/asset_registry.py::
  _scan_dream_presets()` 实际跑一遍确认。

## 要做的

1. 先确认根因：破限预设的存储/加载是否强制要求一个"世界"上下文，还是本来就能
   独立选择/新建（如果本来就能独立操作，那这条单纯是 UI 引导不清楚，改文案/加个
   "不挂世界书，直接写破限"的入口就够，不用动数据模型）。
2. 如果确实耦合：给"破限预设"一个不依赖"世界"选择的独立入口（创建/编辑/选用
   都不强制先建世界），可选挂载一本世界书，但不挂也能单独生效——对齐现实侧
   `reality_jailbreak` 已经在用的独立资产模式，不需要另起一套。
3. 茶茶原话"实现应该很简单"，如果第 1 步确认只是耦合在 UI 流程上（不是数据结构
   问题），这条大概率真的是小活。
