# 108 — 事件总线统一：决策（已拍板）+ 施工单

> 决策人：Fable（2026-07-20）。cc 按施工步骤执行即可，不需要重新讨论方案。
> 前置阅读：`docs/interaction-event-model.md`（三维 envelope v0.1）、`docs/channels.md`。

## 决策

**不做完整事件总线。** 理由：

1. `docs/interaction-event-model.md` §五 已明确把 dispatch router、`kind=tool`、`EventEnvelope` 列为 v0.2+ 预留，且写明"在此列出以防止提前实现"——这是过去的正确判断，当前没有推翻它的新需求。
2. 现在的真实痛点是**可观测性**（不知道哪里在调什么）和**局部卡死**（stage/dream），前者由工单 B5（API 调用总账）解决，后者由 A1/A2 解决。事件总线解决的是"多种 kind 需要统一路由"的问题，而 v0.1 的 LLM 出口只有 message/stimulus 两种 kind，路由器没有客户。
3. 大重构与 C1（目录归拢）同期进行会互相踩脚。

**只做三件小事**（合称"envelope 收敛"，等 C1 完成后施工）：

### 步骤 1 — trust 字段显式化
`PerceiveEvent`（`core/perceive_event.py`）新增显式 `trust: str` 字段（`low_trust` / `high_trust`），默认由 source 推导（现状逻辑固化成代码），调用方可覆写。不改任何 gating 行为——只是把隐式约定变成显式字段，为将来 trust-based gating 留钩子。audit log 同步写入该字段。

### 步骤 2 — trigger→stimulus 命名对齐（仅文档与日志层）
代码文件名/函数名 v0.1 不动（改动面太大收益低）。只做：日志文案与 audit log 的 `kind` 字段统一输出 `stimulus`；`docs/interaction-event-model.md` §3.2 的 DELTA 表更新。

### 步骤 3 — audit 观测端点补全
确认 trigger audit log 有只读观测端点（强制规则 7）；没有则补 `GET /observability/perceive-events`（分页 + 按 gate_result/source 过滤）。这与 B5 的 api-calls 端点放同一 router 文件下。

## 明确不做（防 cc 发散）
- EventEnvelope dataclass / dispatch router / `kind=tool` / `kind=activity` / plugin 系统 / 跨 realm 结构化事件——全部维持 v0.2+ 预留。
- 不改 `perceive_event` 的 gate 语义、不动 Stage/Dream 的入口方式。

## 验收
`pytest --testmon` 通过；audit log 新字段有测试；观测端点可查。整单预计 1 个 commit × 3。
