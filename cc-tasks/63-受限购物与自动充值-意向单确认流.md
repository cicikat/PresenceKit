# Brief 63 · 受限购物 + 自动充值执行（阶段 3，带准入闸门）

> ⛔ **准入条件（全部满足才允许执行本单，否则退回）**：
> 1. Brief 57 台账稳定运行 ≥ 30 天，无 capped 误判、无审计事故；
> 2. 独立小额支付载体就位（单独银行卡小额度 / 平台预充值 / 礼品卡，用户自选并确认）；
> 3. 用户明确拍板开启（本单是全仓第一个"自动花钱"的工单，不默认推进）。
>
> 依赖：57（所有动作过 spend_ledger，这是硬前提）。
> 核心思想：AP2 mandate 三件套（意图单 + 预算封顶 + 审计）本地化实现；
> plan-then-execute 隔离——**读网页的阶段不能花钱，花钱的阶段不能读网页**。

## 1. 自动充值执行（57 的 proposed→confirmed 中间补一段）

1. `config.spend.balance_providers[].auto_execute: false`（默认关，逐 provider 开）。
2. 开启且 provider 有充值 API 的：proposed 行生成后——
   - 金额 ≤ `auto_confirm_cap`（默认 0 = 永远需要确认）→ 直接执行；
   - 否则通知用户，经 `POST /spend/mandate/{id}/confirm`（admin scope）确认后执行。
3. 执行 = 调 provider 充值 API（adapter 按 provider 写，没有 API 的 provider 保持 v1
   通知模式）；结果落 `confirmed` / `failed` 行。**任何执行前必过 `check_budget`**
   （57 的日/月 cap + 白名单，fail-closed）。

## 2. 购买意向单（purchase mandate）

新增台账动作类型 `purchase`，流程严格三段：

**① 选品段（脏环境，零支付权限）**
来源不限（对话中用户说想要 / 角色 web_search 比价）；产出结构化意向单：

```json
{"mandate_id": "pm_...", "item": "...", "max_price": 45.0, "merchant": "...",
 "url": "...", "reason": "...", "expires_at": ..., "status": "draft"}
```

落 `data/runtime/spend/mandates.jsonl`。此段代码路径**不 import 任何执行模块**（静态可验证）。

**② 确认段**
- 通知用户（角色口吻 + 明细）；`POST /spend/mandate/{id}/confirm|reject`。
- 免确认白名单：`config.spend.purchase_whitelist: [{merchant, max_price}]`
  （默认空 = 一切需确认）。
- 超 `expires_at`（默认 48h）未确认 → 自动作废。

**③ 执行段（干净环境）**
- 只读意向单结构化字段，**不读任何网页自由文本**（提示注入的钱路隔离，这是本单
  存在的理由）；执行方式按商家 adapter：优先平台 API/免密小额接口；
  **无 API 的商家 v1 不支持**——执行段没有浏览器。
- 执行前再过一次 `check_budget` + 意向单价格上限比对（实际价 > max_price → 中止 + 通知）。
- 全程落台账；成功后 action_trace 条目（"给你买了…"，角色叙事闭环）。

## 3. 拍板（红线，写进代码注释与测试）

- **不存卡号/支付密码**：schema 无此字段；支付载体的绑定在平台侧完成，
  后端只持有平台 API key（config，gitignored）。
- **执行段无浏览器**：不 import 任何浏览器自动化；国内主流扫码支付流程明确不做
  （风控雷区，见交接文档 §4）。
- 意向单确认 UI 走管理面板（64 补可视化）；聊天内"嗯买吧"这类自然语言确认
  v1 **不做**（确认必须是带鉴权的显式动作，不能被注入伪造）。
- LLM 全程只能产生 `draft` 意向单——`confirm` 与执行入口在 `_TOOL_REGISTRY` 之外，
  模型无路径触达（execute() origin 闸门加负向测试钉死）。

## 4. 测试

1. auto_confirm_cap=0 → 一切充值需确认；≤cap → 自动执行且过 budget。
2. 意向单全生命周期：draft→confirm→执行→confirmed；reject/超时→作废。
3. 实际价超 max_price → 中止 + 通知 + 台账 failed 行。
4. 选品段静态隔离：import 检查（选品模块不引执行模块）。
5. LLM 无确认路径：以 assistant_loop origin 调 confirm 类动作 → 拒绝。
6. 台账写失败 → 一切执行中止（57 fail-closed 契约回归）。
7. `pytest -n auto` 新增测试 + 57 回归。

## 5. 不做什么

- 不做浏览器自动化下单（永久边界，除非另立项并重新过安全评审）。
- 不做比价爬虫（选品用现有 web_search 工具即可）。
- 不做订阅型/周期型自动扣费（一次性购买之外全部超纲）。
