# Brief 117 · 工具循环（memory/mcp/fs）观测面板

写于 2026-07-24。这轮讨论发现：管理面板的"探针观测"(`observe-probe`) 页只读探针
（Path A）快照，命名和实现是一致的、没写错——但**除了这一页，系统里没有对应
Tool Loop（Path C：memory/mcp/fs 类）的统一观测面板**。数据其实都已经存在，只是
散落在好几个地方，没人把它们收进一个跟"探针观测"对等的页面。

## 现状（数据已存在，缺的是页面）

- `data/runtime/memory/{char_id}/{uid}/action_trace.json`：三条工具执行路径
  （探针/意图解析/tool loop）共用的执行痕迹，含 `tool`/`origin`/`args_digest`/
  `result_digest`/`status`/`ts`，环形上限 30 条。
- `api_call_log`：MCP 调用额外记一条，`caller` 固定为 `mcp__{server}__{tool}`。
- MCP 管理页（`admin/static/index.html` 里 `_loadMcpRecentCalls`）已经顺手挂了
  "最近调用记录"，但只覆盖 mcp 这一类，读的是 api_call_log。

## 要做的

新增一个页面（可以叫"工具观测"或者直接扩展 `observe-probe` 成"工具观测"父页 +
tab），按类目分区展示最近的工具执行历史：

- info / desktop：探针（Path A）+ tool loop（Path C）两条来源都可能产生，读
  `action_trace` 按 category 过滤即可。
- memory：目前默认不接入正式对话（见 `docs/tools.md` 该类目备注），但 tool loop
  激活时能看到；同样读 `action_trace`。
- mcp：读 `action_trace` + 复用 MCP 页已有的 `_loadMcpRecentCalls` 逻辑，两边数据
  应该是同源的，不用重复实现，抽出来共用即可。
- fs：读 `action_trace`（`fs_list`/`fs_read` 声明了 `trace_args: ["path"]`，可以
  直接显示路径）。
- system：`device_shutdown`/`device_sleep`/`exit_yandere` 默认关闭且需二次确认，
  执行也会走 `action_trace`，可以放进来但预期条目很少。

后端大概率不需要新接口——`action_trace` 已有读取函数，缺的只是一个按 category
筛选/聚合的小 helper（如果现在没有的话）+ 对应的只读 admin 路由 + 前端页面。跟
"探针观测"一样是**纯只读**，不涉及任何写操作或配置改动。

不紧急，UI 为主的活，跟 116 一样可以独立排期、互不依赖。
