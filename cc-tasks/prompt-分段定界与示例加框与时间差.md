# 工单：Prompt 分段定界符 + mes_example 加框 + 上一条消息时间差

> 仓库 `D:\ai\qq-st-bot`，核心文件 `core/prompt_builder.py`。先看 `AGENTS.md` + `docs/prompt-layers.md`。
> **改的是生成热路径**：每处改完都要 `python tests/run_eval.py`（验层激活），并人工对照检视器抽查 2-3 轮 prompt + 输出，确认无串味/无语气回退。
> 三个问题都来自实测：叶瑄把后续文本误当日记、把语气示例当真实对话、对旧消息有时间错觉。

---

## 决策摘要（你让我定的格式）

**用 XML 式成对标签 `<标签>…</标签>` 作结构定界**，不用只加前缀的 `【】`，也不用 `【\日记】`。理由：
- 缺的就是**闭合**——成对开/闭一次解决"后面文本也算日记"的串味。
- LLM（含 DeepSeek）对 `<tag>…</tag>` 的边界尊重度最高，且真实正文几乎不会出现 `</近期日记>`，不会误伤。
- 可**程序化校验**每层标签是否配平（顺手加个 integrity 断言）。
- 标签名用中文保持可读，内部仍可保留 `【小标题】` 作人类可读抬头。

## 问题 1：每个模块加成对定界符

**现状**：所有模块层只有前缀 `【…】`，无闭合（如 `prompt_builder.py:766` 日记 `【用户的近期日记】\n{diary_context}`）。各层虽是独立 system message，但模型跨消息推理时仍会把紧随的内容并入上一段语义 → 串味。

**改**：给所有**注入外部内容**的模块层，把 content 包进成对标签：
```
<近期日记>
{diary_context}
</近期日记>
```
覆盖范围（逐个 `_layer` 过一遍，`prompt_builder.py` 行号参考）：
- 6d 近期日记(766) / 6e 角色昨日记录(799,810) / mid_term(746) / episodic 片段(731,740) / 6b 相关往事(721) / 3 关系(471) / 群聊上下文(493) / 群聊共享对话(502) / 世界书(691) / 关于用户(648) / 用户客观信息(662) / 待办备忘(678) 等。
- **规则类层**（输出格式硬规则、记忆协议、人格稳定性等 927-1040）也建议包，但这些是稳定指令、串味风险低，可第二优先。
- 标签名取该层语义短名（中文），保持全仓一致；内部原有 `【…】` 抬头可保留。

**配套**：在 `core/integrity_check.py` 或 build 末尾加一个轻量断言——每个被包裹层的 content 满足 `<X>…</X>` 配平（不配平打 WARNING，便于以后新增层时不漏闭合）。

**验收**：检视器里每个模块层 content 都有成对开/闭；构造日记后紧跟对话，叶瑄不再把对话当日记（人工抽查）。

## 问题 2：mes_example 加"仅示例"框 + 与真实对话分界

**现状**：`_parse_mes_example`（`prompt_builder.py:1123`）把示例按 `{{user}}/{{char}}` 拆成 `role:user / role:assistant` 消息（layer `7_mes_example_item`），**和真实历史（layer 9_history, :872）形态完全一致**，且无任何"这是示例"说明 → 模型把语气示例当成真实发生过的对话。

**改**：
1. **示例块前后加框**：在第一条 `7_mes_example_item` 前插一条 system，末条后插一条 system（或把整块包进成对标签）：
   - 前：`<语气示例 note="以下仅为{char_name}的说话风格/语气参考，不是真实发生过的对话，不要当作记忆或事实">`
   - 后：`</语气示例>`
2. **真实对话开始标记**：在 layer 9_history 之前插一条分界 system，如 `<对话记录 note="以下是与用户真实发生的对话">`，history 之后闭合。让模型清楚"示例到此为止，真实对话从这里开始"（对应 SillyTavern 的 `<START>` / "Start a new chat" 语义）。
3. **speaker 区分**：真实历史已是 role 维度区分（user/assistant），API 层面本就分得开；问题主要在"示例 vs 真实"混淆，上面 1+2 已解决。**单聊不必**给每条加名字前缀（避免污染续写风格）；群聊那条共享 transcript 若是裸文本拼接，确认其中已带 speaker 标签（`core/stage` 侧），没有再补。

**验收**：检视器里示例块有明确"仅示例"框、与真实对话有分界标记；叶瑄不再引用示例内容当作真发生过的事（人工抽查，尤其示例里出现的具体情节）。

## 问题 3：给"上一条用户消息"标时间差，防时间错觉

**现状**：short_term 持久化了 `timestamp/speaker_id/turn_id`，但 prompt 只投影 `role/content`（见 `prompt_builder.py:878` 注释）；已有 `2.55_last_seen` 层只给一句笼统的"用户上次说话：{文本}"（来自 `core/presence.get_last_seen_text`）。模型对"这条消息是刚刚还是几小时前"无精确感知 → 时间语义偏移。

**改**：把"距离"显式投影进 prompt：
1. **精确化 2.55 层**：把"用户上次说话"改成相对时间差，如 `用户上一条消息距现在约 3 小时 12 分钟前`（gap 很小则省略或写"刚刚"）。数据用 short_term 最近一条 user turn 的 timestamp 与 now 求差。
2. **大间隔时在消息边界打标**：当 gap 超阈值（建议 >10 分钟）时，在最新用户消息（layer 12, :1064）前加一条简短 system 或在该消息加前缀标记：`<时间提示>距上一条消息已过去约 {间隔}</时间提示>`。gap 小于阈值不注入，避免 token 浪费和啰嗦。
3. 阈值与文案集中成一个小 helper（可放 `core/presence.py`），别散在 prompt_builder 里。

**验收**：隔几小时再发一条 → 检视器里出现"距上一条 X"提示，叶瑄回应不再把旧消息当作刚刚说的；短间隔连聊时不出现该提示。

## 执行顺序与回归

1. **问题 2**（mes_example 加框）——最可能立竿见影，改动局部。
2. **问题 3**（时间差）——局部、独立。
3. **问题 1**（全模块定界符）——面最广，最后做，配 integrity 断言。

每步后：`python tests/run_eval.py` 必跑（确认层激活未被标签改动影响）；用 round4 的 Prompt 检视器抽查 2-3 轮，对照"层内容是否配平、示例是否成框、时间差是否出现"；人工发几轮确认语气无回退。改完同步 `docs/prompt-layers.md`（新增定界符约定 + 示例框 + 时间差层说明）。
