> 映射为**受控表演 spec**，附着在 `message_segments` 的 say 段上下发桌面端，驱动 3D/Live2D

## 1. 协议契约（与客户端文档一致）

`message_segments.segments[]` 元素从 `{type, text}` 扩展为可选携带 `perform`：

```jsonc
{
  "type": "say",
  "text": "才、才没有等你很久呢",
  "perform": {                  // 可选；整体缺失 = 无表演标注
    "expression": "happy",      // neutral|gentle|thinking|happy|sad|surprised|angry|sleepy|yandere | null
    "intensity": 0.7,           // 0~1，缺省 0.6
    "head": "tilt_r",           // nod|shake|tilt_l|tilt_r|dip|null
    "posture": "lean_in",       // lean_in|lean_back|shrink|straighten|null
    "gaze": "away",             // user|away|down|wander|null
    "energy": 0.4               // 0~1，缺省 0.5
  }
}
```

原则：**字段 null = 不覆盖该通道**（客户端回落 mood 基调层）。宁缺毋滥——只在动作/感受文本
有明确信号时才置值，不要给每句强行填满。`content`、msg_id 关联、say-only 投影规则不变。

## 2. 新模块 `core/perform_mapper.py`

```python
async def enrich_say_segments(reply: str, say_segs: list, *, char_id: str) -> list:
    """输入 build_say_segments 的产出，返回同长度 segments（say 段可能多出 perform 键）。
    任何内部异常/超时 → 原样返回输入（fail-open，绝不影响主流程）。"""
```

### 2.1 段落归属（do/feel → say 的挂靠）

内部重新 `parse_narrative_segments(reply)` 取全类型段序列，按顺序扫描：

- 累积遇到的 `do`/`feel` 文本；遇到 `say` 段时，将「累积的 do/feel 文本 + 该 say 文本本身」
  作为该 say 段的映射输入，清空累积。
- 末尾残余的 do/feel（say 之后才出现的动作）追加挂靠到**最后一个** say 段。
- 回复完全没有 say 段时（build_say_segments 已 fallback 为整段 content 单段）：全部文本
  映射为单个 perform 挂在该段上。
- say 段与 `say_segs` 参数按序对齐（两边都来自同一 parser，顺序一致；数量对不上直接 fail-open）。

### 2.2 provider = rules（默认，v1）

纯规则词典，对映射输入文本做关键词/正则匹配 → 填字段。起步词典（CC 可扩充，放模块内常量表）：

| 信号（示例，含常见变体） | 产出 |
|---|---|
| 凑近/靠近/贴近/探身 | posture=lean_in |
| 后仰/退开/往后缩(不含"缩成") | posture=lean_back |
| 缩/蜷/抱膝/埋进 | posture=shrink, energy-=0.2 |
| 挺直/坐直/站直 | posture=straighten |
| 点头/点了点头 | head=nod |
| 摇头/摇了摇头 | head=shake |
| 歪头/侧过头 | head=tilt_r |
| 低头/垂下头/垂眸 | head=dip, gaze=down |
| 看着你/盯着/直视 | gaze=user |
| 移开视线/别过头/看向别处/避开目光 | gaze=away |
| 环顾/张望 | gaze=wander |
| 笑/勾起嘴角/眉眼弯弯 | expression=happy |
| 叹气/垂下肩/黯淡 | expression=sad, energy-=0.2 |
| 瞪大眼/愣住/怔住 | expression=surprised |
| 鼓起脸/瞪了.*一眼/哼 | expression=angry, intensity=0.5 |
| 脸红/耳尖发烫/害羞 | expression=gentle, gaze=away |
| say 文本以"！/!"结尾 | energy+=0.2 |
| say 文本以"……"结尾或含"……"≥2 处 | energy-=0.15 |

- energy 基准 0.5，累加后 clamp 0~1；同字段多信号冲突时**先匹配者胜**。
- 无任何命中 → 该 say 段不带 perform 键（不要输出全 null 的空壳）。
- 词典匹配在 do/feel 文本上优先，say 文本只用于 energy 标点启发和明确的表情词。

### 2.3 provider = llm（v2，config 门控）

- **整条回复一次调用**（不要每段一次）：prompt 给出全部 say 段编号 + 各自挂靠的动作文本 +
  词汇表，要求输出 JSON 数组（编号 → perform 或 null）。严格 schema 校验，非法字段置 null，
  解析失败整体 fail-open。
- 走 `core/llm_client.chat` 新调用类别 `perform`：
  - `_CALL_TIMEOUTS` 增加 `"perform": 10.0`；
  - `core/model_registry.py` 的 `_all_categories` 及 routing 校验处增加 `perform`
    （legacy 合成路径同步），`config.example.yaml` 的 routing_profiles 示例补一行——
    可路由到便宜快模型。
- 外层 `asyncio.wait_for(..., timeout=3.0)` 兜底（可配置）：超时放弃标注直接下发，
  **绝不阻塞 segments 推送超过该时限**。

### 2.4 config

```yaml
performance_mapping:
  enabled: true        # false = enrich 直接透传，行为与现在完全一致
  provider: rules      # rules | llm
  llm_timeout_sec: 3.0
```

`core/config_loader.py` 按现有模式加默认值；缺块 = enabled:true + rules（rules 零成本零延迟）。
