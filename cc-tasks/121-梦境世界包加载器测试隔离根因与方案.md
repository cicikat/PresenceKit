# Brief 121 · 梦境世界包加载器（world/hud/scene/symbolic）测试隔离根因与方案

写于 2026-07-24。这是 CC 全量跑测试后报告的 Category C（17 fail + 3-5 error，跨
`test_dream_world_isolation.py` / `test_dream_integration_fallback.py` /
偶发 `test_dream_body_tension_mechanics.py` / eval）。已查到两个独立、可确认的根因，
方向已经拍板，交给实现即可，不需要再判断。

## 根因一（真根因，导致偶发性）：4 个 loader 模块把 sandbox 路径冻结在 import 时刻

`core/dream/world_loader.py`、`hud_label_loader.py`、`scene_label_loader.py`、
`symbolic_loader.py` 都在模块顶层写了：

```python
_WORLDS_BASE = get_paths().dream_worlds_dir()
```

这行代码只在**模块第一次被 import 时**执行一次，之后 `_WORLDS_BASE` 永远不变——
不管后面 `core.sandbox._instance` 被换成什么。

而 `tests/conftest.py` 有一个 **autouse** fixture `_default_sandbox_guard`，在**每个
测试**开始前都会把 `core.sandbox._instance` 重定向到一个全新的、空的 pytest tmp 目录
（防止漏挂 `sandbox` fixture 的测试误写真实 `data/`）。

`core/data_paths.py::dream_worlds_dir()` 的实现：

```python
def dream_worlds_dir(self) -> Path:
    if self.mode == "test":
        return self._base / "dream_worlds"      # ← 空 tmp 目录，什么都没有
    primary = self.user_dream_worlds_dir()        # userdata/characters/dream/worlds
    return primary if primary.exists() else self.legacy_dream_worlds_dir()
```

`test_dream_world_isolation.py` 里有一批测试（`test_identity_stable_in_world`、
`test_mes_example_isolation`、`test_pronoun_correct_in_world_prompt`、
`test_pronoun_first_person_and_no_reality_label_in_mes_example`、
`test_pronoun_correctness_in_d8_director`）**没有请求 `sandbox` fixture**，也没有
自己往 tmp 目录里塞任何世界包内容——它们的断言（"abo/vampire/cat 世界的 D1 人称
关键词存在"之类）**只有在 `_WORLDS_BASE` 指向真实的 `userdata/characters/dream/worlds`
时才可能通过**。

这些测试能不能通过，完全取决于 `core.dream.world_loader` 这个模块**在当前 worker
进程里第一次被 import 的那个时刻**，`core.sandbox._instance` 恰好是什么——如果这次
import 发生在 pytest 收集阶段（早于任何 autouse fixture 执行），`_instance` 可能还是
生产态默认单例，`_WORLDS_BASE` 就会是真的 `userdata/.../worlds`，测试能过；如果这次
import 是在某个测试函数体内第一次触发（autouse fixture 已经先跑过），`_WORLDS_BASE`
就被冻结成空 tmp 目录，之后**同一 worker 进程里所有测试**都读不到真实内容，全灭。
`pytest -n auto` 是多 worker 并发，每个 worker 各自独立触发一次首次 import，这正是
"两次全量结果不完全一致、这次多炸了 body_tension/eval"这种偶发性的来源——跟收集顺序/
worker 分配相关，不是测试本身逻辑错。

### 方向（已拍板）

1. **去冻结**：4 个模块把 `_WORLDS_BASE = get_paths().dream_worlds_dir()` 这行模块级
   常量，改成每次使用时现算（可以抽一个 `_worlds_base() -> Path: return
   get_paths().dream_worlds_dir()` 私有 helper，各处调用点从 `_WORLDS_BASE` 换成
   `_worlds_base()`）。这不只是为了测试——`core/sandbox.get_paths()` 设计上就是要
   "每次用都现查"，写死成 import 期常量本身就违反 CLAUDE.md Hard Rule 1（所有
   `data/` 路径必须过 `get_paths()`，隐含的前提就是不能自己缓存出一份逃逸的常量）。
2. **去冻结之后，上面那批"读真实内容"的测试会从"偶发过"变成"稳定不过"**——因为
   autouse guard 每次都会先把 `_instance` 切到空 tmp，现算也只会现算出空目录。所以
   要配一个新 fixture（比如叫 `real_dream_worlds`），显式覆盖 autouse guard：

   ```python
   @pytest.fixture
   def real_dream_worlds(monkeypatch):
       """让 dream_worlds_dir() 等 authored-content loader 读取仓库里真实的
       userdata/characters/dream/worlds，覆盖 autouse 的空 tmp 沙盒。
       仅用于显式需要真实世界包内容做身份/人称回归断言的测试。"""
       import core.sandbox as _sandbox
       real_paths = _sandbox.DataPaths(mode=None)   # 非 test 模式，走生产态路径解析
       monkeypatch.setattr(_sandbox, "_instance", real_paths)
   ```

   把这个 fixture 加到上面列出的那 5 个测试的参数列表里（`def
   test_identity_stable_in_world(world_id, real_dream_worlds):` 这样）。

## 根因二（独立的、确定性的 bug，不是偶发）：迁移后遗留的死路径

`tests/test_dream_integration_fallback.py:26`：

```python
_REAL_WORLDS_BASE = _PROJECT_ROOT / "characters" / "dream_worlds"
```

7 月 22 日的资产迁移（`47439c2`）之后，`characters/dream_worlds/` 在磁盘上**已经
不存在**（已确认，`ls` 返回 No such file or directory），真实内容现在在
`userdata/characters/dream/worlds/`。第 80 行 `shutil.copytree(_REAL_WORLDS_BASE /
"_default", ...)` 因此会直接 `FileNotFoundError`——这大概率就是报告里"3-5 error"
（不是 fail 是 error）的来源，跟根因一的偶发 fail 是两件事，凑巧一起被归进同一个
Category C。

### 方向（已拍板）

把 `_REAL_WORLDS_BASE` 从硬编码字符串换成调用生产态 `DataPaths` 现查（避免下次再
迁移一次又要手改这个测试文件）：

```python
import core.sandbox as _sandbox
_REAL_WORLDS_BASE = _sandbox.DataPaths(mode=None).dream_worlds_dir()
```
