## 部分采纳 · 挂 backlog(不立即立项)

- §1.3 "每条消息 5–8 次 LLM":**计数夸大**。第 4–7 项在慢队列异步执行,不占用户等待延迟;真正在关键路径上的是 probe + 主生成 + Path B(将由 35 处理)+ 锁内 detect_emotion(34 处理)。成本论成立,延迟论只对一半。
- §四 20k 字符预算非 token / JSON 失败整段重放 / get_config 每轮几十次 stat / §1.4 全局字典不回收:都对,都便宜,都不急。单用户场景零收益,统一挂 known-issues backlog 区,多用户化之前必须清。

## 拒绝(说明理由)

- **§2.4 函数级 import 治理**:939 处是循环依赖的症状,同意诊断;但处方(重划模块边界)是一次全仓性大手术,风险/收益比在单人维护的项目里不成立。维持现状,新代码不新增即可。
- **§四 重写 _sanitize_assistant_message**:"没人敢碰的代码最该重写"这句话作为格言很漂亮,作为行动指南是错的。Path B 该删是因为**有替代通道**;_sanitize 是承重墙且**没有替代物**,启发式确实丑,但它挡着的是已经发生过的风格坍缩。重写的前置条件是先建风格回归评测集,没有评测集的重写等于拿主观感受赌回归。拒绝立项,条件成熟再议。
- **§3.2 关机能力默认关**:审计漏看了 meta-mode——`desktop`/`system` 类工具已被安全/危险模式闸约束,危险模式默认关、需 hardware scope 显式开、2 小时自动回落,外加 dangerous=True 确认流。诉求("默认关闭、显式开启")**已被现状满足**。
- **§1.5 统一两套串行机制**:诊断对(两个真相来源=认知负债),但现在合并的回归风险远大于收益。降级处理:在 docs/memory.md 并发一节补一张"哪条路径受哪套机制保护"的对照表,把认知负债变成一页纸。

## 总评

审计的局部指控几乎全部属实,值得尊重;它最有价值的一句不是任何一条 bug,而是"测试从安全网变成了防腐层"。但它有一个系统性偏差:**把"多用户压测视角"套在单用户产品上**,导致严重度普遍标高一档(1.1/1.4/1.5 在真实使用场景里都是潜伏问题而非现行问题)。以及它不知道自己审的是谁写的——那些被它归为"对家式增长"的补丁,一半是我签发的。裁定如上,该认的认,该驳的驳。
